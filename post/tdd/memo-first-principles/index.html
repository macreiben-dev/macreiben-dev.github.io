I’m going to write a serie of memos about clean code practices. Here is the first one, the FIRST principles. FIRST principles is a way to determine whether a test is a unit test or not.

The FIRST principles are recommendations to use when you create unit tests - espcially when coding in TDD. FIRST principles makes the tests easier to run.

What does FIRST acronym means ?

FIRST is an acronym that stands for:

  • Fast
  • Isolated
  • Repeatable
  • Self-validating
  • Thorough

Why Fast ?

A unit test is the smallest feedback loop you have on your code. As it is small it is meant to be executed very often. This way we detect behaviour changes quickly.

If a test takes 2 minutes to run, you will eventually get bored and stop using it. Or the test will be executed when you push code to the common repository. The feedback loop time is a core value in test classification.

Long feedback loops is not bad. It allows us to validate other kind of behaviors of our applications. End to end tests targets component communications for example. Acceptance target 3 or 4 component interfactions and so on.

One last thing, test naming and what they are doing depends on your team culture. It means that what I call accpetance tests in this article might be named “end to end” in your team. That’s not a problem as long as we have the same definition behind the words.

They are powerful tools such as NCrunch that can run your tests almost when you are writing it.

Why Isolated ?

Isolated stands for tests should be isolated from one another.

Usually we factorize to reduce the code unit we have to maintain. It reduces the maintenance load on our shoulders. But Factorization generates behavioural coupling.

If two tests rely on the same code unit, each time this code unit changes, the test harness can be broken, or inaccurate (worst). It might not detect breaking changes anymore.

Unit tests are the keeper of the business rules and documentation of your application. The unit tests goal is to alert you whenever a behaviour changes.

This is wh you want your tests to be even more resistant to change than your application.

How do we increase change resistance of a code ? By duplicating code, yes, that very plague we fight against in legacy.

Duplication is bad !

In code things aren’t black or white, I learned it the hard way. I do factorize code in my unit tests. Before I change anything I ask myself the following question :

  • How do I keep the unit tests code’s resistant to change and how do I reduce the code’s maintenance load ?

rather than just “how do I reduce maintenance load”. Just be carefull when you factorize code between unit tests.

Why Repeatable ?

Like the speed of execution, repeatable is at the core of unit testing.

It means a unit test cannot dependend on the states of external dependencies otherwise, it is not a unit test. This way, each test should have its own initial state setup.

Repeatability is achieved when we master the initial state of the tested system and predict its outcome.

Tests have to be deterministic or they can’t be trusted.

Writing test that interact with a database is fine. But you are not writing unit tests.

Why Self validating ?

What if you had to go analyze a big log file to determine whether a certain log line exists to know if the test is pass or fail ?

Again, unit tests are the smallest feedback loop. We want them to be executed fast. We don’t want to lose time analyzing the output.

It wouldn’t be fun and what is not fun is quickly left rotting. For a test it means no being executed anymore and ignored.

The test status have to be loud and clear.

Why Thorough ?

The curse of unit test in real life is … code coverage. Although it is a useful metric, it can corrupt the unit tests aim.

Coverage tells you that some tests are executing a statement. It does not tell you what business rules are actually being tested.

In simple words, 100% coverage is not good enough.

Tests are written to ensure your code supports use cases. Do you have a doubt about how the code would behave under certain condition ? Write a unit test.

Knowing the system behaviour will bring much more serenity than knowing a test pass here.

Tests represent the confidence you have to go to production.
